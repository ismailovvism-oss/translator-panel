# CLAUDE.md — «Вычитка» (сверка перевода)

Инструмент для вычитки перевода: рядом показывает арабский оригинал, основной
русский перевод (редактируемый) и произвольное число справочных вариантов
(в т.ч. от других ИИ). Данные — `.md`-файлы на диске в репозитории перевода,
доступ через File System Access API. Запускается в Chrome/Edge на десктопе
(Linux и Windows).

## Как мы работаем

- Идеи, решения и тексты ТЗ приходят извне как краткие брифы. **Клудод владеет
  всем кодом и структурой репозитория.** Реализуй брифы, не отклоняясь от
  ограничений ниже.
- Если бриф противоречит этим правилам — останови и спроси, не «чини молча».

## Жёсткие ограничения (не нарушать без явного разрешения)

- Ванильные **HTML / CSS / JS**. Никаких фреймворков, никаких сборщиков,
  никакого шага build. Приложение — **один файл** `vychitka.html`.
- **Никаких внешних зависимостей**: ни CDN, ни npm, ни подгружаемых шрифтов.
  Только системные шрифты. Ядро должно работать офлайн.
- Распространяется как статика по HTTPS (Netlify, авто-деплой из репозитория
  приложения); `file://` остаётся рабочим запасным вариантом. Это чистая статика —
  под хостинг в коде ничего менять не нужно.
- Поддержка только десктопных Chrome/Edge (File System Access API). При
  отсутствии API — показывать понятное сообщение, а не падать.

## Рассчитано на нескольких пользователей

Приложение раздаётся другим людям, не только автору:

- Ничего пользовательского не хардкодить. Все настройки (ключ API, раскладка
  слотов, последняя папка) — в `localStorage`/IndexedDB браузера конкретного
  пользователя, **не** в репозитории.
- В репозитории перевода — только контент (`source/`, `translation/`,
  `variants/`) и сам файл приложения. Никаких секретов и личных настроек.
- Интерфейс и сообщения рассчитаны на человека, который видит инструмент впервые.

## Модель данных

Каждый блок имеет **метку** (`метка`) — свободный текст, стабильный идентификатор:
`390-393`, `Глава 1`, `фатава 7`. Метка — это имя файлов блока и основа
спаривания и сортировки. У блока также есть **заголовок** — человекочитаемое
название, которое хранится **внутри** перевода (во фронтматтере), а НЕ в имени
файла.

Три папки в корне репозитория:

- `source/` — арабский оригинал, один файл на блок. Имя = метка: `390-393.md`,
  `Глава 1.md`. Внутри: необязательный комментарий `<!-- ص: N -->` (бледный,
  только в оригинале), структурные заголовки `#`/`##`/`###` (переводимые).
  Редактируемый (правка текста и границ абзацев), сохраняется.
- `translation/` — основной русский перевод, один файл на блок, **единственный
  итоговый артефакт**. Имя = метка: `390-393.md`. Заголовок — во фронтматтере.
  Заготовка:
  ```
  ---
  title: Виды поклонения сердца
  status: todo
  glossary_checked: false
  ---
  ```
- `variants/` — справочные варианты (другие ИИ, ручные правки). На блок может быть
  несколько, поэтому имя = метка + ярлык-источник: `390-393 gemini.md`,
  `Глава 1 мой.md`. Ярлык — свободный текст.

**Спаривание.** `source/<метка>.md` ↔ `translation/<метка>.md` — по совпадению
имени (метки). Вариант относится к блоку, чья метка является самым длинным
префиксом имени файла варианта; остаток (после пробела) = ярлык. (Метка может
содержать пробелы — поэтому именно префикс-совпадение с известными метками, а не
разбор по первому пробелу.)

**Сортировка** списка блоков — по метке по алфавиту (порядок задаётся именем;
менять порядок = менять метку).

Папка определяет роль файла — `source/`, `translation/`, `variants/` не смешивать.

### Правила фронтматтера

- Плоский YAML (`key: value`). При сохранении **сохранять все ключи и их
  порядок**; менять только нужный ключ (`title`, `status`, `glossary_checked`).
  Неизвестные ключи не выкидывать. Ключ `pages` (если есть в старых файлах) —
  сохранять, но он больше не идентификатор.
- Если ключа нет — добавить в конец.
- `title` — человекочитаемый заголовок блока, показывается в панели; правка
  заголовка меняет только это поле, **без переименования файлов**.
- Тело файла нормализуем: срезаем пустые строки в начале/конце; между закрывающим
  `---` и телом — одна пустая строка.
- Оригинал (`source/`) фронтматтера не имеет.

## Окна = слоты

- Слот имеет **роль**: `source` | `translation` | `variant`. У `variant` есть
  ещё **ярлык** (= имя источника/модели).
- Слотов произвольное число; набор слотов — настройка раскладки (не привязана к
  конкретному блоку). При переходе между блоками каждый слот **сам переразрешает**
  файл текущего диапазона: слот `variant:gemini` показывает
  `variants/<диапазон> gemini.md` для активного блока.
- `source` — RTL, редактируемый; заголовки страниц и строка-комментарий рисуются
  бледно/мелко. `translation` и `variant` — LTR, редактируемые. Итогом считается
  только `translation`. У каждого окна — **одинаковая** кнопка «Сохранить» в его
  шапке, пишет только в файл этого окна; в шапке блока — «Сохранить всё».
  Несохранённое помечается фиолетовым (`--unsaved`), не цветом статуса.
- Если файла для слота в текущем блоке нет — пустое поле + «Сохранить» у варианта
  создаёт `variants/<диапазон> <ярлык>.md`. Деградировать мягко, не падать.
- Роль `footnotes` — окно сносок: показывает определения `[^id]:` того окна,
  где последним был фокус (оригинал/перевод/вариант), редактируемо. Своего файла
  нет: правки переписывают хвост определений в ведомом окне, сохранение — файл
  ведомого окна. Курсор на ссылке `[^id]` в тексте прокручивает окно сносок к её
  определению. В черновиках/dirty не участвует (учитывается ведомое окно).

## Поведение (перенесено из V1, не ломать)

- Список блоков: значок статуса (todo / draft / review / verified), диапазон,
  заголовок; сортировка по первому числу диапазона; фильтр по тексту.
- Статусы: `todo → draft → review → verified`. Смена статуса пишет во фронтматтер
  и сразу сохраняет файл (вместе с текущим телом).
- Маркеры `==?...==` (открытые вопросы) — подсветка жёлтым прямо при наборе.
  Реализация: текстовое поле + слой-подложка (родное поведение курсора/undo
  сохраняется). Регэксп живого маркера: `/==\?[\s\S]*?==/g`.
- **Запрет verified при живом `==?...==`**: если в `translation` есть маркер —
  предупредить и не менять статус.
- Отслеживание несохранённого, предупреждение при уходе/переключении блока,
  сохранение по `Ctrl/Cmd+S`.
- **Никогда не терять фронтматтер и чужой текст** при записи. Файлы читать свежими
  при выборе блока (их может менять git или другой инструмент).
- Кодировка — UTF-8 (арабский + русский).

## API-режим (опционально, фаза 2 — только по отдельному брифу)

- Назначение: окно-вариант может само запросить перевод оригинала у выбранной
  модели и положить ответ. Результат сохраняется **тем же** механизмом «Сохранить
  как вариант» → обычный файл в `variants/`. Хранилище не меняется.
- Провайдер по умолчанию — **OpenRouter** (один ключ, один OpenAI-совместимый
  эндпоинт, выбор модели = строка `model`). Не плодить отдельные интеграции под
  каждого провайдера без необходимости.
- **Ключ API**: у каждого пользователя свой. Хранить в `localStorage` его
  браузера, вводится один раз в настройках приложения. НИКОГДА не писать ключ в
  файлы репозитория и не коммитить — репозиторий перевода общий и раздаётся
  другим. Никакой телеметрии и внешних отправок, кроме самого вызова модели.
- При запуске с HTTPS-хостинга (Netlify) вызовы API работают без локального
  сервера — защищённый источник снимает проблему CORS. Локальный сервер нужен,
  только если открываешь как `file://` (там origin «null»).
- Обрабатывать ошибки сети/лимитов мягко (сообщение в интерфейсе, не падение).

## Хранилище за тонким слоем (на будущее, фаза 3 — не сейчас)

Доступ к файлам (список блоков, чтение, запись, создание) должен идти через
**тонкий слой-адаптер**, а не быть размазан по UI. Сейчас адаптер один — локальная
папка через File System Access API. Это нужно, чтобы позже без переписывания
логики слотов/вариантов/статусов добавить **удалённый бэкенд** (для мобильных, где
прямой доступ к файлам недоступен). Наиболее вероятный кандидат — **GitHub API**
(тот же репозиторий, правки = коммиты, работает из браузера, доступ по
персональному токену). Строить это сейчас не нужно — только не мешать этому
архитектурой.
